# 任务简报：翻译 + 出处链接优化

## 项目背景
- 网站：traini.ainchina.com
- 项目目录：europe-train-status/（前端 frontend/，后端 backend/dist/）
- 当前状态：已采集六国运营商约791条disruptions数据

---

## 任务一：翻译优化（核心变更）

### 需求（由5老板明确）
- 目标：将非英语的disruption信息翻译成英语
- **关键约束：不使用外部翻译API（Google/DeepL/OpenAI等）**
- **执行方式：使用agent自身（Q🦐）的翻译能力，在本地环境中逐条翻译**
- 节奏：网站刷新频率不高，agent可以慢慢做，数据量不大
- 已有791条数据已翻译（国家标记[DE]/[AT]等），需要确认是否需要重新走agent翻译
- 新增数据也需要翻译

### 技术方案
- 在数据库中为disruptions表增加翻译字段（translated_title, translated_description, translated_at, translation_engine标记为"agent"）
- agent在本地逐条读取原文→翻译→回写数据库
- 翻译失败时：存原文，翻译字段留空，translation_status标记为"failed"，不阻断其他流程

## 任务二：出处链接

### 需求
- 为每条disruption添加"查看原文"的出处URL
- 用户点击可跳转到运营商官网了解详情

### 关键未知（需Q🦐调研）
5老板提出的合理疑问：**部分carrier数据通过API获取，API响应里可能没有原始网页URL。**

### 调研要求
Q🦐必须调查以下六国运营商的数据来源，明确每种来源是否能拿到出处URL：

1. **DB (Deutsche Bahn, 德国)** - baustellen.html 是静态页面？
2. **ÖBB (奥地利)** - API还是网页？
3. **SBB (瑞士)** - API还是网页？
4. **SNCB/NMBS (比利时)** - API还是网页？
5. **NS (荷兰)** - 之前提到是API endpoint？
6. **SNCF (法国)** - API还是网页？

### 出处URL获取策略（按可行性排序）
1. 如果采集源（API/网页）本身提供permanent link或详情页URL → 直接提取存储
2. 如果API不直接提供URL，但提供了disruption ID，且运营商官网有固定格式的详情页URL → 用模板拼接（如 `https://operator.com/disruptions/{id}`）
3. 如果上述两者都没有 → 退到运营商官网首页或disruptions总览页

### 实施步骤
1. 先调研六国运营商的数据来源和URL可行性
2. 在operator表或新增operator_url_templates表中记录各运营商的URL模板
3. 采集器在抓取时尽量提取/拼接source_url
4. 前端每条disruption旁渲染"查看原文"链接，无URL时隐藏该按钮
