# 劳动仲裁AI MVP项目 - 单聊对话完整梳理

## 对话概述
用户（5老板）与Kimi Claw的单聊，围绕"劳动仲裁纠纷法律服务智能体"的可行性研究与MVP开发。对话从最初的想法验证，经历了12维度深度研究、25000字报告撰写、MVP开发部署、三轮用户反馈迭代，最终进入律师专业反馈修复阶段。

---

## 第一阶段：需求提出与研究启动

### 用户核心诉求
- 创建劳动仲裁法律服务智能体
- 核心洞察：AI对劳动法已很专业，但用户（首次维权的劳动者）无法判断AI说得对不对
- 需要"中间地带"：专业律师背书 + 不只是问答，而是完整可执行方案（流程、证据采集固定等）
- **关键约束**：没有各地社平工资数据、没有经济补偿金计算逻辑、没有判例口径、没有语料库API
- **目标**：最小成本MVP，基本完全依赖大模型能力

### Kimi Claw的响应
启动系统性研究计划：
- Phase 1: 景观扫描
- Phase 2: 维度分解
- Phase 3: 12维度并行深度研究
- Phase 4: 交叉验证
- Phase 5: 置信区间与冲突识别
- Phase 6: 洞察提取
- Phase 7: 报告撰写

## 第四阶段：MVP开发（v1）

### 用户指令
"根据你的分析总结，帮我搭建一个MVP的网页。大模型能力可以不用gemini，用你觉得合适的，或者用kimi也行。把你发现的这些insight都需要融合进去。"

### Kimi Claw的MVP实现
**技术栈**：React + TypeScript + Tailwind CSS + shadcn/ui，纯前端零后端，数据存localStorage

**融合的8个研究洞察**：
1. 证据采集是差异化切入点 → 核心功能：个性化证据清单
2. 律师背书=内容供应链 → 展示"内容审核机制"板块
3. 合规=护城河 → 顶部固定免责声明
4. 地域差异=用户参与点 → 城市选择+数据置信度标签
5. C端+B端变现 → 纯免费，预留律师/企业入口
6. 模型不完美=机会 → 诚实展示不确定性
7. 竞争对手是百度知道 → SEO友好单页结构
8. 工具→内容→社区递进 → 当前专注工具层

**功能模块**：
- 纠纷诊断（9种劳动争议类型）
- 证据清单生成器（核心差异化）
- 赔偿金计算器
- 仲裁流程指引（8步）
- 文书模板库
- 律师资源

**部署URL**：https://rfxmtsrbeu4ye.ok.kimi.link

## 第五阶段：第一轮用户反馈（v1→v2）

### 用户反馈（4条）
1. **选择题流程可跳过** → 应该强制走一遍流程，第二步匹配才能更正确
2. **加班没有地方填时间** → 需要加班时长输入
3. **加班审批记录不是必须的** → 从essential降级为important
4. **输出内容很潦草** → 需要以专业劳动仲裁律师的口吻、经验来输出

### 额外要求
- 主动进行口径数据挖掘（搜索）进行填充
- 不准的数据先不让用户选，写"待上线"
- 尽可能全，覆盖所有目标城市与用户

### Kimi Claw的修复
- 强制引导设计：未选择纠纷类型时，证据清单和赔偿金计算区域显示锁定状态
- 新增加班时长输入：工作日延时/休息日/法定假日，对应1.5/2/3倍
- 加班审批记录降级为"重要"，增加说明"口头指令也算"
- 以专业律师口吻重写全部证据清单："证据描述→获取方式→取证技巧→法律依据"四段式
- 搜索2024-2025官方公报，更新10个城市核心数据
- 每个流程步骤新增法律依据
- 城市数据置信度标签+数据来源标注

## 第六阶段：第二轮用户反馈（v2→v3）

### 用户反馈
"再更新一下，用户遇到的困难可能是多选，需要考虑多选场景的逻辑。比如说即违反劳动合同，又拖欠工资，还付加班工资，社保还有问题"

### Kimi Claw的改造
**从单选 → 多选**：
- 诊断页面：每个纠纷类型前有复选框，可多选
- 证据清单：自动合并多类型证据，去重并按优先级排序
- 赔偿金计算：每个类型分别计算，Tab切换查看，自动汇总总额

**三步流程升级**：
| 步骤 | 改造前 | 改造后 |
|------|--------|--------|
| 纠纷诊断 | 只能选一种 | 可多选，勾选所有符合的选项 |
| 证据清单 | 单一类型清单 | 自动合并多类型全部证据，去重排序 |
| 赔偿金计算 | 单一类型计算 | 每个类型分别计算，自动汇总 |

## 第七阶段：律师专业反馈（v3→当前）

### 用户指令
"这个是律师的使用反馈问题，这些建议都是准确的应该，你可以核验一下。需要做三步：
1. 问题的对应修改
2. 总结这些问题发生的原因，修改相似的问题避免重犯
3. 触类旁通看看是否有可以借鉴的地方，继续改进"

### 律师反馈文件：问题.docx（14条）

Kimi Claw将14条反馈归类为5类：

| 问题类别 | 数量 | 根因 | 修复方案 |
|----------|------|------|----------|
| A. 证据必要性等级不合理 | 3条 | 用"通用模板思维"设计，未考虑案由特殊性 | 全部9个案由重新逐条审核重要性等级 |
| B. 法律细节/时效/口径缺失 | 5条 | 依赖大模型通用知识，缺乏最新司法解释 | 每条证据增加"法律依据"字段 |
| C. 前提条件未声明 | 2条 | 未区分"前置条件"和"后置证据" | 所有案由增加"前提条件"提示 |
| D. 计算功能不完整 | 2条 | 过度简化计算逻辑 | 增加精细化输入项 |
| E. 遗漏高频场景 | 2条 | 产品思维偏向"常见场景"，遗漏灰色地带 | 增加"特殊情形"分支提示 |

### 根因总结（3个）
1. **证据清单用"通用模板"而非"个案思维"** - 被迫离职5项全必须→实际只需四选一
2. **过度依赖大模型通用知识，缺乏最新法律更新** - 不知道2025.9司法解释（二）新规则
3. **未区分"前置条件"和"证据收集"** - 年休假没提"连续满12个月"前提

### 触类旁通的持续改进方向
1. 全案由"前提条件"检查清单
2. 地区差异数据库化（不只是社平工资，还有overtimeBase/severanceBase/socialInsuranceArbitration等）
3. 每条证据的法律依据字段+地区差异标注
4. 特殊情形分支提示
