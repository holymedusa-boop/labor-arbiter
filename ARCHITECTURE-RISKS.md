# 网站架构改进方案风险评估

## 当前方案 vs 改进方案对比

### 方案1: 构建时动态读取 (推荐)
**实现**: 在 Next.js 构建时使用 `fs.readFileSync` 读取 `content/posts/` 目录

**优点**:
- 零额外依赖
- 构建后仍是静态页面，性能无影响
- 写入 markdown 即可自动发布

**风险 ⚠️**:
| 风险 | 严重程度 | 说明 |
|------|---------|------|
| 构建时间增加 | 低 | 文章多了会慢，但目前20篇可忽略 |
| 文件系统依赖 | 中 | 只能在 Node.js 环境构建，Edge Runtime 不支持 |
| 缓存失效 | 低 | Vercel 自动处理，无需担心 |
| 错误处理 | 低 | 需要处理文件不存在、解析错误等情况 |

---

### 方案2: 使用 CMS (Strapi/Sanity)
**实现**: 文章存储在 CMS，通过 API 获取

**优点**:
- 可视化编辑
- 多人协作
- 版本控制

**风险 ⚠️⚠️⚠️** (不推荐当前阶段):
| 风险 | 严重程度 | 说明 |
|------|---------|------|
| 单点故障 | 高 | CMS 宕机 = 网站无法更新 |
| 成本增加 | 高 | Sanity Pro $99/月，Strapi Cloud $9/月起 |
| 网络延迟 | 中 | API 调用影响构建速度 |
| 供应商锁定 | 高 | 迁移数据困难 |
| 复杂度爆炸 | 高 | 需要学习 CMS 系统 |

---

### 方案3: 构建时脚本生成 (平衡方案)
**实现**: 用 Node.js 脚本扫描 `content/posts/` 并生成 `posts-data.js`

**优点**:
- 自动化，无需手动同步
- 零运行时依赖
- 失败可回滚

**风险 ⚠️**:
| 风险 | 严重程度 | 说明 |
|------|---------|------|
| 需要维护脚本 | 低 | 几十行代码，简单 |
| 生成文件冲突 | 低 | 加入 .gitignore 即可 |
| 构建步骤增加 | 低 | 在 `npm run build` 前执行 |

---

## 我的建议

**推荐方案3** (构建时脚本生成)，原因：

1. **风险最低**: 不改变当前架构，只是自动化
2. **成本为零**: 无需额外服务
3. **易于回滚**: 出问题可立即切回手动模式
4. **学习曲线平缓**: 你现有的发布流程不变

### 具体实现思路

```bash
# 发布流程变成:
1. 写入 content/posts/new-article.md
2. npm run build   # 脚本自动生成 posts-data.js
3. git push        # 部署
```

### 潜在问题及对策

| 问题 | 对策 |
|------|------|
| 脚本出错导致构建失败 | 添加 try-catch，出错时不阻断构建 |
| 文章格式不一致 | 统一 frontmatter 模板 |
| 图片路径问题 | 继续使用 Unsplash 外链，避免本地图片 |

---

## 不建议现在改进的情况

如果出现以下情况，**保持现状**更好：
- 发布频率不高（每周1-2篇）
- 当前流程已经熟练
- 不想投入时间改代码
- 对现有性能满意

## 总结

| 方案 | 风险等级 | 推荐度 |
|------|---------|--------|
| 保持现状 | 低 | ⭐⭐⭐ (如果你接受手动同步) |
| 构建时脚本生成 | 极低 | ⭐⭐⭐⭐⭐ (最佳平衡) |
| 构建时动态读取 | 低 | ⭐⭐⭐⭐ |
| 使用 CMS | 高 | ⭐ (过度设计) |

**核心建议**: 如果要改进，选择「构建时脚本生成」，风险最小，收益明显。
